# Context

## 1. Overview

"Context" in this system refers to everything that is assembled and sent to an LLM before it generates a response. The quality and relevance of the context directly determines the quality of the LLM's output.

A key design decision is that **each LLM call builds its own context independently** — there is no shared global context object. The context for a security check is minimal and clean; the context for a chat response is rich and multi-layered; the context for slide generation is entirely different from the chat context.

```
  ┌──────────────────────────────────────────────────────────────────┐
  │                     Workflow LLM Calls                           │
  │                                                                  │
  │  Step 1: Security Check                                          │
  │  ┌──────────────────────────────────────┐                        │
  │  │ System: SECURITY_CHECK_PROMPT        │                        │
  │  │ User:   user_input                   │                        │
  │  └──────────────────────────────────────┘                        │
  │                                                                  │
  │  Step 2: Route and Answer                                        │
  │  ┌──────────────────────────────────────┐  ┌──────────────────┐  │
  │  │ System: Instructions + Tools +       │  │ tools= parameter │  │
  │  │         User Facts + History +       │  │ (OpenAI function │  │
  │  │         Summary                      │  │  calling schema) │  │
  │  │ User:   user_input                   │  └──────────────────┘  │
  │  └──────────────────────────────────────┘                        │
  │                                                                  │
  │  Step 3: Slide Generation                                        │
  │  ┌──────────────────────────────────────┐                        │
  │  │ System: Slide Instructions +         │                        │
  │  │         History + Summary +          │                        │
  │  │         Previous Slide HTML          │                        │
  │  │ User:   "User Request: {user_input}" │                        │
  │  └──────────────────────────────────────┘                        │
  └──────────────────────────────────────────────────────────────────┘
```

## 2. Context Components

These are the building blocks that get assembled into context. Not all components appear in every LLM call.

### 2.1. System Instructions (Prompts)

The foundation of every LLM call. System prompts define how the LLM should behave, what format to output, what rules to follow.

| Prompt constant | Used in | Purpose |
|----------------|---------|---------|
| `SECURITY_CHECK_PROMPT` | Step 1 | Classify user input as SAFE or EXPLOIT |
| `ROUTER_ANSWER_PROMPT` | Step 2 | Decide intent (GENERAL/PPTX), answer questions, enforce JSON output |
| `TOOL_BEST_PRACTICES` | Step 2 | Rules for when to use / not use tools |
| `SLIDE_GENERATION_PROMPT` | Step 3 | HTML slide design rules, structure, output requirements |
| `PRESENTATION_INTENT_PROMPT` | Step 3 sub-call | Classify slide action (CREATE_NEW/EDIT_SPECIFIC/EDIT_ACTIVE) |
| `SUMMARY_INITIAL_PROMPT` | Summary sub-call | Generate first summary from raw messages |
| `SUMMARY_UPDATE_PROMPT` | Summary sub-call | Merge old summary with new messages |

### 2.2. Tool Definitions

Tool information reaches the LLM through **two separate channels** in Step 2:

**Channel 1 — Injected into system prompt** (text, visible as instructions):

```
AVAILABLE TOOLS:
- get_weather: Get current weather information for a city. Use when user asks about weather.
- get_stock_price: Get stock price by ticker symbol. Use when user asks about stock prices.
- add_user_fact: Save user's personal information (key-value). Use when user requests to remember info.
- update_user_fact: Update a saved user fact by key. Use when user requests to modify their saved information.
- delete_user_fact: Forget/Delete a saved user fact by key. Use when user requests to forget/remove their saved information.
- extract_url_content: Extract content from a website URL for summarization. Use when user provides a URL and requests to read/summarize it.
```

This is generated by `registry.get_tool_instructions()` using each tool's `name` and `summary` fields.

**Channel 2 — Passed as `tools=` parameter** (structured JSON schema, used by OpenAI function calling):

```python
openai_tools = [tool.metadata.to_openai_tool() for tool in tools]
resp = await llm.achat(messages, tools=openai_tools)
```

Each tool is converted to OpenAI's function calling schema using the tool's `name` and `description` fields (more detailed than `summary`). This is what enables the LLM to actually call tools — the schema defines the function signature the LLM must follow.

### 2.3. User Facts

User facts are personal information the user has explicitly asked the system to remember (e.g., "Remember that my name is Bao", "Save that I work at Acme Corp").

**Source:** `user_facts` table, filtered by `user_id`

**Format when injected:**

```
USER FACTS (Information about the user):
- name: Bao
- city: Hanoi
- company: Acme Corp
```

Generated by the private helper `_get_user_facts_text(user_id, db)` inside `context_service.py`, called from `build_chat_context`. If the user has no saved facts, this component is skipped entirely — no empty section is added.

**Scope:** User facts persist **across all conversations** of the same user. A fact saved in one conversation is available in all future conversations.

### 2.4. Chat History (Short-term Memory)

Recent messages from the current conversation, giving the LLM awareness of what was discussed before the current message.

**Source pipeline — two-stage filtering:**

```
  DB: messages table
  WHERE conversation_id = X
    AND is_in_working_memory = TRUE   ← Stage 1: only active messages
  ORDER BY created_at ASC
         │
         ▼
  ChatMemoryBuffer(token_limit=2000)  ← Stage 2: token budget enforcement
  memory.put(each message)
  history = memory.get()              ← returns only messages that fit within 2000 tokens
                                         (oldest messages dropped if over limit)
```

**Format when injected:**

```
===== RECENT CHAT HISTORY =====
user: What is the weather in Hanoi?
assistant: {"intent": "GENERAL", "answer": "The weather in Hanoi is 30°C and partly sunny."}
user: What about Tokyo?
assistant: {"intent": "GENERAL", "answer": "The weather in Tokyo is 22°C and cloudy."}
```

> Note: assistant messages contain the raw JSON responses, not just the text. This is because all messages are saved as-is to the database.

**Scope:** Only messages from the **current conversation**. Messages from other conversations are never included.

### 2.5. Conversation Summary (Long-term Compressed Memory)

When chat history grows beyond the token limit, older messages are summarized and removed from working memory. The summary is appended to the context **after** the chat history.

**Source:** `conversation_summaries` table — exactly **1 row per conversation** (upserted on each summary update)

**Format when injected:**

```
===== CONVERSATION SUMMARY =====
The user asked about weather in several cities including Hanoi and Tokyo. The assistant provided
current weather information using the weather tool. The user then asked about stock prices for
AAPL and TSLA...
```

**Relationship with chat history:**
- Chat history = recent turns (token-limited, `is_in_working_memory = TRUE`)
- Summary = compressed record of older turns (`is_in_working_memory = FALSE`)
- Both can be present simultaneously; summary always appears after history

**Scope:** Per conversation. Each conversation has its own summary.

### 2.6. Previous Slide Content

Only present in the Slide Generation context (Step 3). Contains the HTML content of an existing presentation being edited.

**Source:** `presentation_pages` table via `load_presentation(target_presentation_id)`

**Conditional loading — amount injected depends on edit scope:**

| Edit scope | What gets injected |
|-----------|-------------------|
| `CREATE_NEW` | Nothing — no previous slides |
| Single-page edit (`target_page_number != null`) | Only the target page's `page_title` + `html_content` |
| Full presentation edit | All pages: each page's `page_title` + `html_content` |

Each `html_content` is a complete HTML document (~1280×720px slide). For a 5-page presentation, injecting all pages can add tens of kilobytes to the prompt.

## 3. Context per Workflow Step

### 3.1. Step 1 — Security Check

The most minimal context in the system. Intentionally contains no history, facts, or memory — a clean, isolated classification call.

**LLM used:** `llm_security` — a separate, lightweight model with `temperature=0` for deterministic classification.

**Why no history or context:** Adding history could give a prompt-injecting user a way to smuggle context that influences the security classification.

### 3.2. Step 2 — Route and Answer

The richest context in the system. The system prompt is built by concatenating components in a fixed order:

```
  ┌──────────────────────────────────────────────────┐
  │                  System Prompt                   │
  │                                                  │
  │  ① ROUTER_ANSWER_PROMPT                            ← always present
  │    Intent rules, tool rules, output format       │
  │                                                  │
  │  ② Tool instructions                               ← always present (if tools enabled)
  │    AVAILABLE TOOLS:                              │
  │    - tool_name: one-line summary                 │
  │    ...                                           │
  │                                                  │
  │  ③ TOOL_BEST_PRACTICES                             ← always present
  │    When to use / not use tools                   │
  │                                                  │
  │  ④ USER FACTS (if any)                             ← conditional
  │    - key: value                                  │
  │    - key: value                                  │
  │                                                  │
  │  ⑤ ===== RECENT CHAT HISTORY ===== (if any)        ← conditional
  │    user: ...                                     │
  │    assistant: ...                                │
  │                                                  │
  │  ⑥ ===== CONVERSATION SUMMARY ===== (if any)       ← conditional
  │    Compressed record of older turns              │
  │                                                  │
  └──────────────────────────────────────────────────┘
  ┌─────────────────────────────────────────────────┐
  │  User Message:  user_input                      │
  └─────────────────────────────────────────────────┘

  + tools= parameter (passed separately, not in prompt text)
  ┌─────────────────────────────────────────────────┐
  │ tools: [ OpenAI function schemas for each tool ]│
  └─────────────────────────────────────────────────┘
```

**Ordering rationale:**
- Instructions (①②③) come first — the LLM reads these before any data, ensuring rules are applied to the context that follows
- User facts (④) come before history — personal context is a stable background, while history is dynamic
- History (⑤) comes before summary (⑥) — recent turns are more immediately relevant than the older compressed record

**Conditional sections:** Components ④⑤⑥ are only appended when they have content. An empty `USER FACTS` block or `CHAT HISTORY` header is never written if there is nothing to show.

**LLM used:** `llm` — the main model, supports tool calling.

### 3.3. Step 3 — Slide Generation

The system prompt is built from scratch — it does **not** reuse or extend the Step 2 system prompt. Tools are not used in this step.

```
  ┌──────────────────────────────────────────────────┐
  │                  System Prompt                   │
  │                                                  │
  │  ① SLIDE_GENERATION_PROMPT                         ← always present
  │    Design rules, slide structure,                │
  │    output format requirements                    │
  │                                                  │
  │  ② ===== RECENT CHAT HISTORY ===== (if any)        ← conditional
  │    Gives context about what the user wanted      │
  │                                                  │
  │  ③ ===== CONVERSATION SUMMARY ===== (if any)       ← conditional
  │                                                  │
  │  ④ Previous slide content (if EDIT)                ← conditional
  │    ┌ Single-page edit:                           │
  │    │  "===== PREVIOUS SLIDE - Page N ====="      │
  │    │  Page Title: ...                            │
  │    │  HTML Content: <full HTML of page N only>   │
  │    │  "Edit ONLY Page N. Backend will merge."    │
  │    └ Full presentation edit:                     │
  │       "===== PREVIOUS SLIDE - All N Pages ===="  │
  │       Page 1: title + full HTML                  │
  │       Page 2: title + full HTML                  │
  │       ...                                        │
  │                                                  │
  │  ⑤ Final action instruction                        ← always present
  │    "Create a NEW presentation..." OR             │
  │    "EDIT Page N..." OR                           │
  │    "EDIT the entire presentation..."             │
  │                                                  │
  └──────────────────────────────────────────────────┘
  ┌─────────────────────────────────────────────────┐
  │  User Message:  "User Request: {user_input}"    │
  └─────────────────────────────────────────────────┘
```

**No `tools=` parameter** — this step uses `response_format: json_schema` instead, which is only compatible when tools are not present.

**LLM used:** `llm` — same main model as Step 2.

### 3.4. Presentation Intent Detection Sub-call

A dedicated LLM call within Step 3 to classify what the user wants to do before any generation starts.

```
  ┌──────────────────────────────────────────────────┐
  │                  System Prompt                   │
  │                                                  │
  │  PRESENTATION_INTENT_PROMPT                      │
  │  Rules for CREATE_NEW / EDIT_SPECIFIC /          │
  │  EDIT_ACTIVE classification                      │
  │                                                  │
  └──────────────────────────────────────────────────┘
  ┌──────────────────────────────────────────────────┐
  │  User Message:                                   │
  │  ===== AVAILABLE PRESENTATIONS =====             │
  │  Presentation 1 (ACTIVE):                        │
  │    - ID: <uuid>                                  │
  │    - Topic: Machine Learning                     │
  │    - Pages: 5                                    │
  │  Presentation 2:                                 │
  │    - ID: <uuid>                                  │
  │    - Topic: World History                        │
  │    - Pages: 4                                    │
  │                                                  │
  │  ===== USER REQUEST =====                        │
  │  Edit page 2 of the slide about history          │
  │                                                  │
  │  ===== YOUR TASK =====                           │
  │  Analyze and output: action, target_id, page_num │
  └──────────────────────────────────────────────────┘
```

Note that the presentation list (metadata only — no HTML) is placed in the **user message**, not the system prompt. This keeps the system prompt stable and cacheable while the dynamic data varies per request.

### 3.5. Summary LLM Sub-call

Called automatically when `ChatMemoryBuffer` detects that messages have been truncated. Context depends on whether a summary already exists.

**First-time summary (no existing summary):**

```
  System: SUMMARY_INITIAL_PROMPT
          "Briefly summarize the key points of the conversation..."

  User:   "Please summarize the following conversation:
           User: Hello, what's the weather in Hanoi?
           Assistant: The weather in Hanoi is 30°C...
           User: What about stocks?
           ..."
```

**Incremental summary update (summary already exists):**

```
  System: SUMMARY_UPDATE_PROMPT
          "Combine old summary with new conversation without repetition..."

  User:   "Previous summary:
           The user asked about weather and stocks...

           New conversation:
           User: Remember my name is Bao
           Assistant: Saved: name = Bao
           ...

           Please create a new summary combining both sections."
```

Messages are formatted by `format_messages_for_summary()`:
```
User: Hello, what's the weather in Hanoi?
Assistant: The weather in Hanoi is 30°C and partly sunny.
User: What about Tokyo?
```

**LLM used:** `llm_summary` — a separate model instance dedicated to summarization tasks.

## 4. What Is NOT in Context

Understanding what is excluded from context is as important as understanding what is included.

| Excluded | Reason |
|----------|--------|
| Messages with `is_in_working_memory = FALSE` | These have been summarized; their content is compressed into the conversation summary |
| Slide HTML stored in message `metadata` | When a PPTX message is added to `ChatMemoryBuffer`, only the `answer` text is stored — not the HTML pages. This prevents slide HTML from bloating future prompts |
| Messages from other conversations | Each conversation's history is isolated — `load_chat_history` always filters by `conversation_id` |
| User facts from other users | `load_user_facts` always filters by `user_id` — complete tenant isolation |
| Presentations from other conversations | `load_presentation` joins through `Conversation.user_id` to verify ownership |
| Raw tool execution results in history | Tool messages exist during the tool calling loop but are never saved to the database — they are ephemeral within a single request |
